#What is Kipp?

Most inference engines try to run everything. Kipp deliberately runs one thing, and aims to run it well enough that you can read the whole engine in an afternoon.

📜 Readable by design

The reference forward pass stays linear and model-specific — a contributor should be able to read src/kipp.c top to bottom and follow one token from embeddings to logits.

⚡ Native all the way down

C11 core, a narrow Objective-C bridge with Metal kernels for Apple Silicon, and isolated CUDA kernels for NVIDIA. Python and shell are reserved for tooling and never enter the inference process.

🧭 One model family, deeply

Kipp is not a generic runner. It will support one explicitly selected model family and a small, documented set of checkpoints — in the lineage of whisper.cpp and antirez/ds4, not llama.cpp or vLLM.

🔒 Correctness first

The CPU path is the reference. Every GPU backend must match its outputs within documented tolerances, on real hardware, before it is called working.

💾 mmap-backed loading

Model weights are mapped, not eagerly copied into RAM, wherever the selected weight format permits it.

🌐 Small serving surface

A minimal, non-streaming OpenAI Completions server runs today, one request at a time on loopback — scheduling and batching only arrive with correctness tests.

#Repository tour

The codebase is intentionally small. Every path below exists today and is implemented through the CPU oracle, KV sessions, the validated Metal backend, native sampling, and the minimal Completions server.

Kipp/
├─ src/
│  ├─ kipp.c            ← core engine: model loading, tensors, the reference forward pass
│  ├─ kipp_server.c     ← non-streaming OpenAI Completions server (loopback)
│  ├─ metal/kipp_metal.m ← the only Objective-C in the project; Metal bridge for Apple GPUs
│  └─ cuda/kipp_cuda.cu  ← CUDA kernels + host launch code, isolated from other backends
├─ tests/
│  ├─ kipp_test.c       ← native test harness
│  └─ test-vectors/     ← pinned model-generated reference vectors
├─ tools/               ← Python/shell tooling only — never part of inference
│  ├─ download_model.sh ← fetch + verify model artifacts
│  ├─ convert_to_gguf.py← convert source weights into Kipp's supported format
│  └─ bench.py          ← reproducible benchmark harness
├─ docs/                ← design docs + this site
├─ context/             ← research notes on reference codebases
├─ AGENT.md             ← contribution constraints (humans and AI agents alike)
└─ Makefile             ← explicit per-backend build targets

#Building

Each backend has an explicit target — there is no auto-detection, by design:

make cpu          # CPU reference oracle + CLI
make metal        # Apple Silicon backend   — validated on Apple M5
make server-metal # OpenAI Completions server on the Metal backend
make cuda-generic # generic NVIDIA backend  — unvalidated until Phase 4 hardware
make cuda-spark   # tuned NVIDIA config     — unvalidated until Phase 4 hardware

The CPU target compiles with -std=c11 -Wall -Wextra -Wpedantic and is expected to stay warning-clean.
