# Kipp Architecture

This document is intentionally model-agnostic until Kipp's first target model
family is selected.

## Design constraints

- Implement one model family deeply rather than building a generic runner.
- Keep the inference path native and readable, with C as the center of gravity.
- Use mmap-backed model loading where the selected format permits it.
- Keep CPU, Metal, and CUDA behavior behind explicit boundaries.
- Establish correctness on the CPU reference path before optimizing kernels.
- Keep Python and shell outside the inference process.

## Planned components

### Model and weights

`src/kipp.c` will parse the selected model's metadata and tensor layout,
validate supported variants, and map weights without unnecessary copies.
Exact formats, quantization schemes, and tokenizer behavior remain undecided.

### Forward pass

The reference forward pass will remain linear and model-specific so a reader
can follow one token from embeddings through logits. Backend dispatch should
not obscure the model's computation graph.

### Backends

- **CPU:** correctness and debugging reference, written in portable C11.
- **Metal:** Apple Silicon kernels with Objective-C confined to
  `src/metal/kipp_metal.m`.
- **CUDA:** NVIDIA kernels and launch code isolated under `src/cuda/`.
- **ROCm/HIP:** explicitly deferred to a separate, community-maintained branch.

Backend interfaces, supported hardware, and kernel placement will be fixed
after model selection.

### Runtime and serving

`src/kipp_server.c` will eventually provide small OpenAI- and
Anthropic-compatible endpoints. Scheduling, KV-cache layout, batching, and
streaming semantics remain design work; none is implemented by this scaffold.

## Correctness boundaries

Reference test vectors will be generated by tooling and stored or reproduced
under `tests/test-vectors/`. Backend outputs must be compared against the CPU
reference path with tolerances documented per numeric format. A backend is not
considered working until it has been run on its actual hardware.

## Open decisions

- First model family and exact supported checkpoints
- Weight container and quantization formats
- Tokenizer ownership and chat-template behavior
- KV-cache representation and capacity policy
- Initial API surface and concurrency model
- Minimum supported Apple and NVIDIA hardware
